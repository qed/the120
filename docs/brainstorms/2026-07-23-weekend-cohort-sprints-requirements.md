# Weekend Cohort Sprints — Founders Weekend Feature Brief

## Problem Frame

Founders Weekend (https://founders.school/founders-weekend) is a free 48-hour intensive for ages 9–16: Friday ideation, Saturday building and customer outreach, Sunday shipping and presentations. Five cities in summer 2026 — Chicago (Jul 24–26), San Francisco (Jul 31–Aug 2), Los Angeles (Aug 7–9), Boston (Aug 21–23), Hamptons (Aug 28–30) — 30 spots each, solo founders or teams of up to 3. Alpha Guides work the floor with iPads and laptops. Students bring their work to a Guide, the Guide checks it against the curriculum, and the whole room watches a projected cohort dashboard fill up.

Today none of that touches software. This brief specifies **Weekend Cohort Sprints** (internal name only): the Guide check-in surface and the projected cohort dashboard, built inside the **First Profit** app. First Profit is the product name for the app previously specified as "The Path" (`docs/brainstorms/2026-07-21-the-path-app-requirements.md` and its plans). The Guide surface will eventually live at **`/fp/fw`**, but it ships initially under **`/path/fw`** — the whole-app `/path` → `/fp` rename is a separate, later step in the same plan (FW-R30).

The core loop is deliberately tiny: a student walks up, the Guide navigates **student → stage → criterion → unit task** in a few taps, reads the task against what the kid did, and taps **Checkmark** or **Not yet**. The dashboard updates within seconds. That is the whole product. Everything that makes The Path deep — evidence, reviews, celebrations, parents — is explicitly out (see Roadmap).

This is a delta brief in the style of the Path brainstorm: where it is silent, the Path requirements, decisions (D15–D27), and data model stand. Where they conflict, this document wins for the FW surface.

## Redesign Pass (2026-07-23, second pass)

Four decisions are **settled** (user-decided, no longer open):

- **SETTLED — Boston (Aug 21–23) is the software debut, full v1.** Chicago (starts tomorrow), SF, and LA run on paper. Hamptons (Aug 28–30) is the second software event. See FW-D14.
- **SETTLED — No per-child XP number on the projected board.** The grid shows each student's cells filling in; cohort-total XP is the only XP figure displayed. The parent brief's no-cross-student-comparison boundary stays intact — no amendment needed. See FW-R27/FW-D13.
- **SETTLED — Offline check-in capture is in v1**, as a scoped IndexedDB queue for check-in writes, with the guide surface itself navigable offline (roster + task tree — a queue you cannot navigate to is no queue). See FW-R31/FW-D15.
- **SETTLED — Batch multi-select check-in is in v1.** One guide action can check a task for up to three students (the team case); teams still do not exist in the data. See FW-R22/FW-D16.

Three changes remain **PROPOSED** — flagged inline, each needs an accept/reject:

- **PROPOSED-1 — Returning-student matching at both entry points** (reshaped after the security re-review). Quick-create gets a privacy-safe "same kid?" confirm; the bulk importer runs the same match per row so pre-loaded returners resolve to their existing account instead of silently duplicating. Honest framing: 2026 walk-in cross-city returner volume is plausibly ~zero — this is cheap duplicate-prevention insurance for the permanent record and the `.fw` namespace, and it is the mechanism that makes the one-account Success Criterion true at all. Amends FW-R9/FW-R12.
- **PROPOSED-2 — First Dollar counter as a persistent co-hero on the board.** A running "🔔 N first dollars" tally beside cohort XP. No stated success criterion depends on it — the case is narrative (the event is *named* for this moment) — which is why it is deliberately the lightest proposal and an early release valve. Amends FW-R26.
- **PROPOSED-3 — The bulk importer doubles as the paper-backfill path, consent-paired.** Chicago/SF/LA's paper records enter the system afterward, cohort-stamped and backfill-marked — but only alongside a consent mechanism for those families, who registered before any notice existed (see FW-R12 for the two options). Value restated honestly: the durable one-system record and PROPOSED-1's matching substrate — not anything visible at the Boston debut, and not timing/not-yet analytics (paper carries neither). Amends FW-R12.

Everything else from the review stands, with sharpened rationale where the pass attacked it: stage-weighted XP is **kept** (the case for flat XP was made and rejected — see FW-R27), FW-D8/FW-D12 are ratified with the alternatives now killed on code evidence, FW-D9's mechanism is settled, FW-D2's liability is materially narrowed (auth account ≠ mailbox), and FW-D4's analytics claim is resharpened to what a single-timestamp log genuinely yields.

## Requirements

**Roles and Guide accounts**

- FW-R1. The account taxonomy is five roles: **student, parent, staff, admin, guide**. **Student, parent, and guide** are expressed as grants (role + scope), continuing D24's role-grant model — one human can hold several. **Staff and admin are *not* grants** (the `path_role_grants` CHECK does not include them): they are the separate `app_metadata.role === "admin"` claim + active `staff` row, bridged to implicit guide powers per FW-D9.
- FW-R2. Guides are the **same guide role** The Path defines (R25/D25). What a guide can see and do depends entirely on which cohort(s) they are scoped to. An FW cohort simply has no evidence, reviews, or countersigns in it, so a guide granted into one gets check-in powers and the board, nothing more.
- FW-R3. **Guides have no CRM access.** The `/crm` boundary (`requireStaff()`, proxy rules) must reject a guide session. Conversely, **staff and admin accounts automatically carry guide powers** — a staff member can do anything a guide can, in any FW cohort, without an explicit guide grant (mechanism and scoping in FW-D9).
- FW-R4. Guide accounts are created two ways: (a) **pre-provisioned** by staff/admin ahead of the event — the `createUser` *mechanics* mirror the CRM staff pattern (a guide never carries the admin claim, FW-D9), but **credentials are issued via an invite-link flow** reusing the hashed, single-use, expiring token pattern of `app/path/lib/actions/invite.ts`: the guide sets their own password on first open, and no staff-distributed plaintext password ever exists; and (b) an **event join-code** that lets a guide self-register on their own device on site. The join-code is per cohort, staff-issued, **high-entropy, attempt-throttled (reuse the repo's rate-limit rules), and auto-expiring with the cohort's event window** — manual revocation is the backstop, not the mechanism (a code overheard in a loud public room otherwise grants check-in/undo power over every minor in the cohort). With only Boston and Hamptons as software events and a known guide roster, planning should confirm the join-code path is needed for 2026 at all (it is also the first release valve, FW-D14).
- FW-R5. Guides sign in with email + password on iPad or laptop. The surface at `/fp/fw` (`/path/fw` until the rename) is responsive; iPad Safari is the primary target, laptop browsers equally supported.

**Student accounts (Guide-provisioned, tracking-only)**

- FW-R6. There is **no student sign-in and no student-facing surface** in this version. Students never touch the system; Guides operate it for them.
- FW-R7. A guide (or staff/admin) creates a student by entering **first name, last name, and grade band** (3–5, 6–8, 9–12). On-site creation must be fast enough to do with a kid standing there: three fields, done.
- FW-R8. Each student gets a **real, dormant Supabase auth account** with the address `firstname.lastname.fw@the120.school`. The address pattern is a firm requirement: `the120.school` is a real Google-Workspace domain with MX records, and these addresses are a deliberate future contact and continuity channel. The accounts are dormant — no password issued, no sign-in path, and **no auth mail is ever triggered for them in this version**. `email_confirm: true` is mandatory on every `createUser` call, per the documented hard requirement (docs/solutions/integration-issues/supabase-admin-createuser-non-deliverable-email-requires-email-confirm-2026-07-21.md). **The invariant is enforced by *mechanism*, not developer discipline: FW accounts are provisioned password-less, and every auth-mail-capable call (recovery, invite, magic-link, email-change) is gated behind a refusal of `*.fw@the120.school`, with a regression test.** `email_confirm: true` suppresses only the *signup confirmation* — it is necessary but not sufficient. Liability note sharpened in FW-D2: an auth account on the domain does **not** create a mailbox; whether a stray mail would actually deliver depends on Workspace routing, which is checked in week 1 (Dependencies).
- FW-R9. **Name collisions auto-resolve by appending a number** (`maya.chen2.fw@…`), silently, at the email-minting layer. The address derivation never blocks the on-site flow and the guide never sees the address. **PROPOSED-1 (accept/reject)** adds one *identity* step above the *email* layer — reshaped after the security re-review, which found the original design leaked cross-cohort PII and created an enumeration oracle:
  - **Quick-create confirm.** If the normalized (first, last) matches an existing FW student **in a cohort the acting guide is scoped to (or a shared cohort)**, the guide sees the full match card — "Maya Chen — band 6–8 — this cohort. Same student? **[Same student] [New student]**". If the only matches are **cross-cohort**, the guide sees a minimal signal — "A student with this name is already in the system — confirm with staff before creating a new record" — with no band, city, or date disclosed; staff (whose bridge spans FW cohorts, FW-D9) resolve the match. *Same student* adds a `path_cohort_members` row for this weekend, no new account; *New student* proceeds exactly as today, silent suffix and all — the flow never hard-blocks.
  - **The lookup is rate-limited and logged** — repeated quick-creates must not become an oracle answering "did this named child attend an FW event, where, and when."
  - **Importer matching.** The bulk importer (FW-R12) runs the same normalized lookup per row: an exact (first, last, band) match to an existing FW student writes a membership row instead of minting an account; ambiguous matches (same name, different band; multiple candidates) land on a **staff-reviewable exception list**, never silently either way. Without this, every Boston→Hamptons returner duplicates through the *primary* entry path and the one-account Success Criterion is false at both 2026 software events.
  - **Honest rationale:** the five 2026 cities are distinct metros and only Boston→Hamptons are adjacent, so walk-in cross-city returner volume this summer is plausibly ~zero. The case is not this summer's walk-ins; it is cheap duplicate-prevention insurance for the permanent record and the `.fw` namespace (`maya.chen.fw` and `maya.chen2.fw` being the same child degrades the CRM asset FW-D2 exists to build), at the cost of one indexed lookup, one conditional dialog, and one import branch. Risk: a guide links the wrong same-named kid — bounded by the same-cohort-only detail card and the staff exception path, and repairable (events are cohort-stamped and actor-attributed). Recommend accept.
- FW-R10. The student's **grade band is stored on the FW student record**, set by the guide at creation. FW students have no `public.children` roster row, so band cannot be derived from grade as The Path does (R31) — it is entered directly. Task views resolve band variants from this stored band; a task without a variant shows its base text per the curriculum's own rule.
- FW-R11. Every FW student is **pinned to the 2026-27 program version at provisioning** (D27 applies unchanged).
- FW-R12. Students enter the system by two paths: (a) **staff bulk import** — staff/admin paste or import the registration roster into a cohort before the weekend, provisioning accounts in one pass (with per-row returner matching if PROPOSED-1 is accepted); and (b) the guide's **on-site quick-create** (FW-R7) for walk-ins and anyone missed. Bulk import is **in scope for Boston** (it is the debut's roster path, no longer the first cut — FW-D14). **PROPOSED-3 (accept/reject):** the same importer accepts a **paper-backfill mode** — a CSV of (first, last, band, checked task ids) from a paper weekend (Chicago/SF/LA), provisioning students and writing their check-ins as events stamped with that cohort, `actor` = the entering staff member, and a `backfill` marker in the note. Restated honestly after re-review:
  - **Cost:** one import variant, **plus the unstated real cost — transcription**: ~90 students' handwritten records keyed to task ids, entered by a named owner on a deadline inside the Aug 10–17 window. That labor lands in the pre-Boston crunch or never.
  - **Value:** the summer's full record in one system, the CRM hook, and the matching substrate that lets a Chicago kid pre-registered for Boston resolve to one account (PROPOSED-1). **Not** anything visible at the Boston debut (the board is per-cohort), and **not** timing/not-yet analytics — paper carries check-state only, and backfilled events are excluded from timing analytics via the marker.
  - **Consent pairing (required for accept):** these families registered before any notice existed and cannot exercise a deletion right they don't know about. Either (i) a **retroactive notice email to the paper weekends' registrants** mirroring the Boston draft (sent to the *parents* who registered, before accounts are minted), or (ii) backfill lands as **non-auth profile rows** (name/band/events, no `createUser`), with account minting deferred until consent or a return visit. Recommend accept with (i).
  - **Sequencing:** if accepted, backfill is entered **before the Boston roster import runs**, so returner matching has something to match against. If cut, record that PROPOSED-1's matching covers software-event returners only.

**Content and navigation**

- FW-R13. The FW surface exposes the **full 2026-27 package — all 125 unit tasks** — not a weekend subset. Rationale: a student may start one weekend, continue at another, or join a different 120 program; the checkmark record lives on the student, not the event, so the content space must be the shared one. No new content is authored for this feature.
- FW-R14. Guide navigation is **student → stage (Sell, Build, Validate, Grow, Scale) → criterion → unit task**, optimized for reaching any task in a handful of taps. The student list is the cohort roster; search-by-name within the cohort is required at realistic cohort size (~30–90 including guides' typos).
- FW-R15. The unit task view renders, top to bottom: the **title as the headline with the check-off control beside it** (near/right of the title); below, the **body** for guides who need the full description; below that, the **Done-when line** — band-resolved per FW-R10. Headline = `title`, subhead = `body`, decision line = `doneWhen`. **FW reading rule (added after re-review):** the done-when lines were authored for home study, and some clauses are literally unsatisfiable at the event — 1.2.3 requires "a parent playing the buyer," 1.2.4's g3_5 variant requires a parent physically present, 1.2.5 requires a photo "in the Founder File," while FW has no parents in the loop and no evidence capture. The FW surface adopts an explicit rule, surfaced once on the guide UI: **parent-role clauses read as guide-role; Founder-File / photo / record clauses read as "the guide has seen it."** Without the rule, guides improvise silently per-guide, and not-yet analytics partly measure clause inapplicability rather than task difficulty (FW-D4). A spot-audit of the weekend's hot Stage-1–2 tasks against this rule is a before-event item.
- FW-R16. **No gating.** A guide can navigate to and check off any task in any order. The locked → available chain of the Path state machine does not constrain FW check-ins; the FW write path must bypass or sidestep it deliberately (FW-D12).

**Check-in mechanics**

- FW-R17. The task view offers exactly two actions: **Not yet** and **Checkmark**. Not yet is a **recorded state**, not a neutral default — it writes an event and shows on the dashboard as attempted-and-back-to-work, matching the curriculum's "nothing is failed, only not done yet" register.
- FW-R18. **Any guide can check in any student within their cohort.** There is no student-to-guide assignment.
- FW-R19. **Both decisions can be undone** by any guide in the cohort (fat-fingers on an iPad at a loud event are a certainty — and a mis-tapped **Not yet** would otherwise permanently write a false struggling signal into the record and the struggle-time analytics, so it is as reversible as a mis-tapped checkmark). The undo affordance is the task view's own check-off control acting as a **toggle** (re-open the task, tap to revert) — not a separate screen. An undo returns the task to its prior **unchecked** state via a fixed target — **`locked`**, uniform with never-touched siblings (see FW-D12); it does **not** route through the Path `revoke` transition (which lands on `not_yet`, a *meaningful* "attempted" signal). XP falls because it is recomputed from the current verified rows, not a stored counter being decremented; nothing is erased from the log (FW-D12). Undo is always per-student per-task, even when the checkmark landed via a batch (FW-D16). Replayed *offline* undos carry an extra guard (FW-D15) so a stale queue cannot revert another guide's live check-in.
- FW-R20. **Every yes and every change is timestamped and logged** — actor, role, student, task, cohort, from-state, to-state, timestamp — following the `path_task_events` pattern. The event log, not the current state, is the source of truth. Stated purpose: future analytics on which unit tasks are easier, harder, faster, slower, and which are blockers needing more support (what the log can and cannot answer is stated honestly in FW-D4 and Success Criteria). Guide attribution is **captured now, surfaced later** (Roadmap) — v1 shows no "checked by" anywhere. Two event-side fields the Path schema does not have (the `submitted_at` split lives on progress rows the FW path bypasses): a **client capture time** on queued-replay events (FW-R31), and a **shared action id** stamped on every event one guide action writes (FW-D16's once-per-action bell is not derivable without it). Both join the schema step (Outstanding Questions).
- FW-R21. Every check-in event is **stamped with the cohort it happened at**, so a student's record reads across weekends ("1.1.1–1.1.3 at Chicago, 1.1.4–1.2.2 at Boston") and results are trackable per weekend, not just per student.
- FW-R22. **Individuals in the data; teams at the surface.** Teams of up to 3 exist at the event but not in the data model — a team clearing a task together is N individual check-ins and N individual records. **Settled 2026-07-23:** the task view carries a **batch affordance** — the guide can add up to two teammates from the cohort roster and apply one Checkmark/Not-yet to all selected, writing one event per student in a single action stamped with a shared action id (FW-D16). The First Dollar moment fires **once per action id**: a team's shared sale rings the bell once, while each teammate's record still gets 1.2.4. A team entity, team names, and team display-grouping remain Roadmap.

**Cohorts and events**

- FW-R23. **Each city-weekend is its own cohort** (e.g. `fw-chicago-2026`). Cohorts carry a **kind flag** (`path` | `fw`) — the mechanism FW-D9's bridge and the no-evidence surface both key on. Guides are granted into the cohort(s) they work. Only **staff/admin create cohorts**.
- FW-R24. Cohort membership is **many-weekends-per-student**: a returning student keeps one account and one progress record, and is added to each new weekend's cohort. Membership rows carry a joined-at timestamp, giving the roster of every specific weekend (with FW-R21 attributing the results).

**The cohort dashboard**

- FW-R25. The projected dashboard is reached by an **unguessable public token URL per cohort** — any laptop can drive the projector with zero login friction. Token rows carry both an **`expires_at` derived from the cohort's event window** (Sunday 6 PM + a grace buffer) and a `revoked_at` for manual kill — auto-expiry is the mechanism, staff revocation the backstop, because "staff remember to revoke" is not a control (the reused invite pattern pairs hashed tokens with a TTL for the same reason). The screen shows minors' **first name + last initial** ("Maya C.", FW-D11) and progress and is treated as semi-public within the room; the token is the only thing keeping it in the room.
- FW-R26. The dashboard displays, for one cohort:
  - the **big grid** — every student down the side, stages/criteria across, cells filling in as checkmarks land. **No per-child numeric score of any kind appears** (settled 2026-07-23 — see FW-D13);
  - **cohort rollups** — totals per stage, share of cohort past each criterion;
  - a **live ticker** of recent check-ins ("Maya C. ✓ Sell 1.2");
  - **not-yet activity** as an in-the-arena signal (attempted, back to work);
  - a **First Dollar celebration moment** — when **task 1.2.4 ("Ask until one yes", done-when: *money from a non-family customer is in hand and the sale is logged*)** is checked, the screen celebrates big, echoing the event's bell-ringing ritual (FW-D10) — celebrations grouped by action id (one bell per team, FW-R22);
  - **PROPOSED-2 (accept/reject):** a persistent **First Dollar counter** ("🔔 14 first dollars") beside the cohort XP hero — the running count of this weekend's first dollars (students whose 1.2.4 verify is cohort-stamped here; undo-safe because it is recomputed, not incremented). Honest framing: **no stated success criterion depends on this** — the case is narrative (the event is *named* for the moment; a transient celebration alone buries the trophy stat) plus the register split (XP is the always-moving pulse, hundreds of moves per weekend; the counter is the rare, huge one, ~dozens). That is why it is the lightest proposal and an early release valve. Recommend accept.
- FW-R27. The **hero metric is cohort-total XP**. XP is earned **per unit task checked**, weighted by stage: a Stage 1 task = 1 XP, Stage 2 = 2 XP, … Stage 5 = 5 XP. Not-yet earns nothing; an undo lowers the total (recomputed from current verified rows, not a stored counter — FW-D13). Per-child XP is **not displayed anywhere on the board** (settled 2026-07-23). **The stage weighting was challenged this pass and is kept.** The load-bearing argument is durability: FW is the first surface to *define* XP for First Profit, and one product-wide definition (later stages worth more) beats a weekend-local one that would fork the number later. The supporting math, stated carefully: a first-time cohort works overwhelmingly in Stages 1–2, where weighted and flat XP track closely — not identically (the event's own Saturday is building, which is Stage 2 territory under no-gating), but close enough that the weighting neither distorts the weekend nor inflates it; the 4–5×-weighted stages are unreachable and therefore inert. Validate against Boston's real distribution before reusing the claim.
- FW-R28. The dashboard **auto-updates by polling every 3–5 seconds**, with the ticker and XP animating on change so it feels live. No realtime/push infrastructure in v1.
- FW-R29. The grid view carries an event title, e.g. **"Founders Weekend — Boston 2026"**. The name "Weekend Cohort Sprints" is purely internal and appears nowhere user-facing.

**Offline (settled 2026-07-23 — promoted from Roadmap)**

- FW-R31. **Check-in capture tolerates a venue-wifi outage — including reaching the next student.** Two halves, both required (the re-review showed the first is worthless without the second):
  - **Writes queue.** Checkmark, Not-yet, and undo actions taken while offline queue locally (IndexedDB) and sync when the device reconnects, replaying in capture order with client-generated idempotency ids so at-least-once replay yields exactly-once events. Replay re-runs full server-side authorization at sync time (a revoked guide's queue does not land), and replayed undos are guarded per FW-D15.
  - **The guide surface stays navigable offline.** The cohort roster (cached at session start, refreshed opportunistically) and the full task tree — titles, bodies, done-whens, search — remain reachable without a network. The content package is static and bundleable; without this, the guide can toggle the task view currently on screen and nothing else, the second student in line is unreachable, and the clipboard returns — the exact failure FW-D15 was promoted to prevent. The Path service worker's never-cache-navigations discipline does not extend unmodified to the `/path/fw` app shell; planning owns the caching shape.
  - Out of scope offline: student quick-create (account minting is server-side) and the dashboard (its stale-poll indicator is the honest signal).

**Naming and routing**

- FW-R30. The app formerly specified as The Path is product-named **First Profit** and **moves from `/path` to `/fp`**. This rename applies to the whole app and should be reflected in the design brief and subsequent planning. **Sequencing: the FW surface ships *first* under `/path/fw` so the event feature is never gated on the rename; the whole-app `/path` → `/fp` move is a distinct *later* step in the same plan (FW's routes move to `/fp/fw` with it) — with the calendar now known, it lands after Hamptons (Aug 30) and before the Path cohort starts (Sept 19).** The Guide surface lives at **`/fp/fw`** (`/path/fw` until the rename step); the dashboard token URL lives under the same tree (e.g. `/fp/fw/board/<token>`).

## Success Criteria

- A guide who has never seen the system checks in a real student — find student, reach the right unit task, read Done-when, tap — in well under a minute, on an iPad, in a loud room.
- A three-kid team's shared clear is **one** find-tap cycle (batch check-in), not three, and the First Dollar bell rings once for the team.
- On-site student creation (first, last, band) takes seconds and never blocks on an email collision.
- A checkmark appears on the projected dashboard within ~5 seconds of the tap, XP animating.
- A mis-tap — checkmark **or** Not-yet — is reversed by any guide in the cohort, and the event log retains both the mistake and the reversal with actors and timestamps.
- **A 20-minute wifi outage during Saturday capture loses nothing:** guides keep navigating between students and tapping, queued check-ins land when connectivity returns with their capture times preserved, and the board catches up. (This is what backs the adoption criterion below.)
- A student who attends two weekends has one account, one progress record, and per-weekend attribution — **delivered by PROPOSED-1's matching at both the importer and quick-create, if accepted.** If rejected, duplicates accrue through *every* entry path (the importer included) until the Roadmap merge tool, and this criterion honestly shrinks to per-weekend attribution only.
- No path exists by which a guide session reaches `/crm`, another cohort's students, or any student mutation outside their cohort scope. (If PROPOSED-1 is accepted, the one deliberate carve-out is the quick-create lookup: on an exact normalized-name match it may disclose, across cohort lines, only "this name already exists — confirm with staff" — never band, city, or date — and the lookup is rate-limited and logged.)
- **Adoption holds through a full Saturday afternoon:** guides keep capturing in-system rather than reverting to a clipboard, so the projected board stays a live mirror of the room and does not silently go stale — the failure mode no other criterion would catch.
- After the summer, the event log alone can answer, **for the software-event cohorts (Boston, Hamptons)**: which tasks drew the most **not-yets**, per-task **clear rates**, per-cohort/per-weekend **throughput**, per-task **attempt-to-clear durations** (the `not_yet → verified` gap on the same student-task — a true struggle-time signal), and per-student **check-in cadence** (inter-check-in gaps). Backfilled paper weekends contribute roster and check-state only and are excluded from timing, not-yet, and clear-rate analytics via the backfill marker. The log **cannot** answer time-on-task for tasks cleared on the first attempt — FW records one timestamp per decision, and that limit is accepted (the rejected "started" tap is documented in FW-D4).

## Scope Boundaries — Roadmap / V2

Everything below is deliberately out of v1 and parked here so it is not forgotten:

- **Evidence capture** — no photos, video, audio, or documents; the guide's in-person judgment is the verification.
- **Review ceremonies** — no criterion/phase reviews, no countersign.
- **Celebrations engine** — the dashboard ticker, XP, and First Dollar moment (plus the PROPOSED-2 counter, if accepted) are the only celebration surfaces.
- **Parent accounts and parent visibility** of FW progress.
- **Student sign-in** or any student-facing surface for FW students.
- **Notifications** — no email or push of any kind.
- **Export** — no CSV/Founder-File-style export of FW data.
- **Team entity** — team names, team records, display-grouping. (The batch check-in *surface* is in v1 per FW-R22; the data stays individual.)

## Key Decisions

- **FW-D1 — Full catalog, no weekend package.** FW exposes all 125 tasks of 2026-27 rather than a purpose-built weekend curriculum, because students continue across weekends and into other 120 programs, and the record must live on the student in the shared content space. Cost accepted: deeper navigation for guides, mitigated by FW-R14's tap-optimized drill-down — and the content is read under the FW reading rule (FW-R15), because home-study done-when clauses (parents, Founder File) are otherwise unsatisfiable at the event.
- **FW-D2 — Real dormant accounts on a real, deliverable domain.** Chosen over profile-only rows for continuity, and over `.invalid` addresses because the address pattern is itself a product asset (future contact/data channel through Google Workspace). Consequence: the no-auth-mail rule (FW-R8) is a hard invariant, not a convenience. **Sharpened this pass — the liability is narrower than the review framed it:** a Supabase auth account on the domain does **not** create a Workspace mailbox. Unless Workspace routes the address (a user, alias, or catch-all), mail to `maya.chen.fw@the120.school` bounces — which is the *ideal* resting state: the namespace is reserved (the asset) without a live mailbox in a minor's name (the liability). The week-1 check (Dependencies) verifies no catch-all swallows `*.fw@` today; the recommended posture is to route nothing until the CRM hook is actually built, deferring the mailbox liability to the moment the asset is used — under whatever consent the registration language (below) established. One honesty note: the Workspace-side check is point-in-time — a catch-all added in a later summer silently arms every dormant address, so the retention/ops policy must own that invariant going forward.
- **FW-D3 — Five-role taxonomy with staff/admin ⊇ guide.** Staff and admin automatically hold guide powers; guides hold nothing of staff's. One guide role serves both The Path and FW, differentiated purely by cohort scope.
- **FW-D4 — The event log is the product's second customer.** Timestamps on every yes and every change are a stated requirement in service of task-level analytics, so the log schema is designed for querying (task, cohort, gaps between events), not just audit. **Resharpened this pass:** a single timestamp per decision yields more than the review credited — not-yet counts, clear rates, throughput, **attempt-to-clear durations** (`not_yet → verified` on the same student-task), and per-student cadence — and less than the original claim: no time-on-task for first-attempt clears. A "started" tap was evaluated and **rejected**: guides interact only at the decision moment (they cannot know when a kid at a table began), so a third tap would add friction to a two-action surface and produce inconsistent garbage data, the worst of both. The claim shrinks to what the log delivers; the offline queue's client capture time (FW-R20/R31) keeps even outage-era gaps honest. One behavioral dependency the re-review named: the struggle analytics exist only if guides actually *record* not-yets rather than saying them out loud and backing out — the dry-run brief states that not-yet taps are wanted data, and dry-run not-yet volume is the canary.
- **FW-D5 — No gating, deliberately against the Path engine's grain.** The locked→available chain protects a year-long journey; it would sabotage a 48-hour sprint. The FW write path bypasses it, and the reconciliation story for later Path conversion is planning's problem, flagged in FW-D12.
- **FW-D6 — Token-URL dashboard.** Zero-login projection wins over auth for an on-site screen; revocable, auto-expiring per-event tokens bound the exposure of minors' first names and progress.
- **FW-D7 — `/path` → `/fp`.** The app is First Profit everywhere; the rename lands with this build, sequenced after Hamptons and before Sept 19 (FW-R30).

*Added 2026-07-23 after codebase verification and three product decisions; ratified and extended in the redesign pass:*

- **FW-D8 — One student table; `child_id` relaxed to nullable (ratified this pass; planning implements).** The lowest-blast-radius shape for the biggest forced decision — and the redesign pass killed both alternatives on code evidence. (a) *Minting `children` rows for FW students* is blocked: `children.parent_id` is NOT NULL with CASCADE from `parents` (20260709200000_initial_schema.sql), so every FW walk-in would need a fabricated parent account — worse identity debt than a nullable FK, and it pollutes the CRM roster with hundreds of non-lead rows. (b) *A 1:1 sidecar table for name/band* doesn't avoid the real change — `child_id` must become nullable regardless (an FW profile has no roster row) — and it moves the XOR integrity check across tables where Postgres can't express it as a plain CHECK. So: `path_task_progress`, `path_task_events`, `path_reviews`, and every grant scope key off `path_student_profiles.id`, not `child_id`; keeping FW students in the same table preserves the single progress/event graph that FW-R13/R21/R24 require. `child_id` becomes nullable (NULLs are distinct in the unique index, so multiple FW rows are fine), the table gains `first_name`/`last_name`/`band`, and a CHECK enforces the XOR: a Path row has `child_id` + null name/band; an FW row has null `child_id` + non-null name/band. If PROPOSED-1 is accepted, add an index on the normalized name pair for the returning-student lookup. Name-read sites that today join `public.children` fall back to the profile's own columns for FW rows — a smaller blast radius than it looks, since FW students never sign in (FW-R6) and have no family dashboard.
- **FW-D9 — Staff/admin ⊇ guide is a bridge between two authz systems, not a new grant (mechanism settled this pass).** Implementation: the **impure action shell** — not the pure resolvers — resolves admin-as-guide. `resolveActorRole` receives only `{ grants, target }` and `resolvePathAccess`'s `SessionLike` has no `app_metadata`, and widening those tested Path signatures for an FW concern pollutes the core; instead, the FW actions check the session's `app_metadata.role === "admin"` claim **and re-verify the active `staff` row** (`resolveStaffAccess`'s revocation rule — the stale-JWT case) before treating the caller as a guide, and do so **only for cohorts flagged `kind = 'fw'`** (FW-R23). The bridge therefore can never grant read or verify over a real Path family's evidence — an admin acting on a *Path* cohort's minors still requires an explicit, D26-style audited guide grant, out of scope here. The mirror — guides out of `/crm` — needs no new guard: `resolveProxyOutcome` already rewrites every non-admin session off `/crm` to 404; planning adds a test pinning it. **A guide account must carry no admin claim** (FW-R4's "mirror the CRM pattern" is `createUser` mechanics only) — a test asserts it. The FW write path resolves the acting guide's cohort match against the **authoritative** membership row, never a client-supplied id (the IDOR seam `access-rules.ts` warns about).
- **FW-D10 — First Dollar = task 1.2.4.** "Ask until one yes" — done-when *money from a non-family customer is in hand and the sale is logged* — the literal first dollar and the bell-ringing moment. Designated as a single task id (a `FW_FIRST_DOLLAR_TASK` constant in the mold of `STAGE_MOMENT_CRITERIA`). Fires on the check-in of 1.2.4, **not** on criterion 1.2 clearing (task 1.2.5, "Deliver, thank, and log," lands moments later and is the record-keeping close, not the money moment). For a batched team check-in containing 1.2.4, the moment fires **once per action id** (FW-R22). The re-review composed two of the brief's own premises — mis-taps are a certainty, and this is the one output undo cannot recompute (the room already rang the bell) — so the task view carries a guard on exactly this task; the shape (one-tap confirm vs a one-poll-cycle celebration delay) is the deferred-design list's one open call, with the confirm recommended.
- **FW-D11 — Dashboard shows first name + last initial.** "Maya C." on the grid and ticker — identifiable to a student's own people, resistant to a stranger fully naming a minor on a projected screen. Depends on `last_name` on the FW student record (FW-D8). Duplicate "First L." pairs at ~30–90 students are plausible; the board-side tiebreaker is on the deferred-design list.
- **FW-D12 — FW check-ins write the same rows via a dedicated, cascade-free transition (ratified this pass; planning implements).** FW writes the same `path_task_progress` / `path_task_events` rows — so XP, analytics, audit, and later Path continuity all come from the one event stream — through a sanctioned FW write path that (a) is legal from any current state, (b) hardcodes `checkmark→verified` / `not-yet→not_yet` / `undo→locked` — the fixed unchecked target (not a caller-supplied "prior" — that would forfeit `move_path_task`'s forged-target guarantee, and routing through `revoke` would wrongly land on the meaningful `not_yet`; `locked` keeps an undone task uniform with never-touched siblings and the reconciliation input space to one shape), (c) stamps `snapshot_band` from the FW student's stored band on the checkmark (the Path `unlock` arm that normally freezes band is skipped, so without this FW rows freeze base-text variants), (d) does **not** run the `verify` cascade (unlock-next + `path_maybe_open_review`), (e) appends a `path_task_events` row exactly as today — cohort-stamped (FW-R21), `actor_role = 'adult'` (the events CHECK is `student|adult|system`, and `resolveActorRole` already maps a guide grant to "adult"; guide identity is the actor uuid + cohort stamp — no CHECK widening), carrying the client capture time for queued replays (FW-R31) and the shared action id (FW-R20), and (f) accepts a client idempotency id so an at-least-once replay yields exactly-once events. **FW provisioning eagerly materializes all 125 progress rows (`locked`), mirroring Path Unit 6** — `move_path_task`'s contract never upserts and treats a missing row as an anomaly, and the FW path honors the same contract. Predecessors stay `locked`. Reconciliation on later Path conversion is a one-time "rationalize" pass that is **larger than a parenthetical**: it must decide whether verified-with-locked-predecessor is a legal intermediate, unlock every predecessor of a verified task, re-derive reviews, and — because `path_reviews.opened_by` is NOT NULL — open any retroactive review under a **documented system/reconciliation sentinel actor** (never a fabricated adult, which would corrupt R6's verification record). It must run **atomically before the student's first Path render**, or the journey shows an illegal graph. Flagged for planning, not solved here.
- **FW-D13 — No per-child score, no ordering by progress (settled 2026-07-23).** The Path's own cohort board carries a hard, tested rule (T3 plan Decision 1): no cross-student comparison; a ranked table of children is a leaderboard whatever it is called. The FW board inherits it **intact** — the redesign pass put the per-child XP column to the user and it was **dropped**, so no narrowing amendment exists: the grid shows each student's cells filling in (their own position, not a numeric comparison), row order is fixed (name; the ticker orders by check-in recency), and **cohort-total XP is the only XP figure on screen**. This is what reconciles "Founders Weekend is about making XP go up" with the no-leaderboard boundary: the number that climbs is the room's total, full stop.
- **FW-D14 — Boston (Aug 21–23) is the software debut, full v1 (settled 2026-07-23).** The dates are real and close: Chicago starts the day after this pass (Jul 24–26), SF is 8 days out, LA is 15. Rather than debut a thin slice at LA or a heroic one at SF, the decision is a **full v1 with a comfortable runway**: build complete and staff-dry-run by ~Aug 17, debut at Boston, patch week, second event at Hamptons (Aug 28–30). Chicago, SF, and LA run on paper — recoverable into the system via PROPOSED-3 if accepted. Consequence: bulk import is in scope (it is Boston's roster path). **Stated plainly, because the re-review asked for it:** this build plus the rename consumes the entire remaining runway to the Sept 19 Path cohort, whose own outstanding work (T2's Founder File export, the unwritten wisdom deck, D22's paper-to-app migration) sits behind it — the bet that FW feeds Path enrollment is being made deliberately, not by omission. **Release valves if the timeline slips, in cut order, each with the criterion it relaxes named** (a valve pulled under pressure must not silently fail an acceptance bar):
  1. Join-code self-registration — relaxes nothing (pre-provisioned guides cover 2026).
  2. PROPOSED-3 backfill — relaxes nothing at Boston; forfeits the summer record and PROPOSED-1's substrate for paper-weekend returners.
  3. PROPOSED-2 counter — relaxes nothing; the transient First Dollar celebration remains.
  4. Batch multi-select — **explicitly relaxes the one-cycle team criterion** (a 3-kid clear becomes three cycles).
  5. Offline queue + offline navigation — **relaxes the outage criterion**; last resort before the never-cut core.
  PROPOSED-1 is not a valve of its own: its quick-create half rides the quick-create step, and its importer half rides the importer — cutting the importer to quick-create-only relaxes the one-account criterion, and the cut order must say so if exercised. Never cut: the loop, the board, the schema.
- **FW-D15 — Offline check-in in v1: queue + navigable surface (settled 2026-07-23; scope corrected in re-review).** The brief's own top-flagged risk was venue wifi dying on a Saturday afternoon — guides revert to clipboard, the board silently goes stale, and the adoption criterion fails in exactly the way no other criterion catches. "Operational only (hotspots)" was hope, not mitigation. Promoted because the FW write is a *tiny, idempotent, no-media* payload — drastically simpler than the Path's evidence sync (no uploads, no TUS, no rebase): an IndexedDB queue drained on foreground/online signals (the proven Unit 11 pattern), client-UUID idempotency, capture-order replay per student-task. The re-review corrected the original writes-only scope: **the queue is worthless if the surface it feeds cannot be navigated offline**, so FW-R31 requires the roster and task tree offline too. **The undo-replay guard (the one conflict CAS does not settle):** a queued offline undo (expected-from `verified`) replayed after another guide's legitimate checkmark would *win* the CAS and silently revert a check-in the room already saw. Rule: (a) a checkmark/undo pair for the same student-task **cancels within the local queue** before replay — the offline fat-finger resolves to a no-op; (b) a surviving replayed undo applies only if the check-in it reverts was written by **the same actor** (mirroring `move_path_task`'s revoke verifier-guard, for replays only — online undo stays open to any guide per FW-R19). Device posture: IndexedDB assumes non-private Safari on staff-managed iPads (a dry-run checklist item); the queue is cleared on sign-out.
- **FW-D16 — Batch check-in without a team entity (settled 2026-07-23).** The event's real unit is a team of up to 3; forcing three full find-tap cycles per shared clear tripled the loop's cost and made the First Dollar moment ambiguous. The task view gains an "add teammates" multi-select (up to 2, from the cohort roster); one Checkmark/Not-yet applies to all selected, writing one **individual** event per student in a single action — no team entity, undo stays per-student (FW-R19). The one data-model touch the re-review forced: every event in the action carries a **shared action id** (the idempotency id's group key), because a polling board otherwise cannot distinguish one team's 1.2.4 from three coincident first dollars in the same 3–5 s window — and ringing the bell wrong (robbing two kids or triple-ringing one sale) is the most visible failure in the product. First Dollar fires once per action id; each teammate's record still gets 1.2.4, and each still counts in the PROPOSED-2 tally (the count reads "founders who made their first dollar," true per kid).
