# feat: The 120 Admin CRM (`/crm`)

## Overview

Build the staff-only CRM described in `artifacts/crm-design-brief.md`: a `/crm` route group for two staff users (peter@the120.school, ethan@the120.school) that makes the live funnel visible (pipeline derived from `parents`/`children`/`deposits` truth), absorbs roadmap ticket S5 (dossier review queue with payment visibility), delivers the GTM weekly-metrics dashboard, and closes the loop with a deterministic co-pilot and a CASL-gated send composer. The pattern language is ported from alphahub (`C:\Users\pkupe\Aardvark\alphahub` — reference only, nothing imported directly); the skin is The 120 design system (`artifacts/The 120 Design Handoff/`).

Delivery is phased exactly as the brief's §13: **P1 See the truth → P2 Run the week → P3 Close the loop.** Units below are grouped by phase.

## Problem Frame

Two operators are running an 8-week GTM sprint (`artifacts/gtm-8-week-sprint.md`) toward 48–55 deposits by Sept 1. The funnel already writes truth to Supabase (signups, dossiers, Stripe deposits, attribution), but that truth is invisible — there is no staff surface at all (verified: no middleware, no role checks, no admin routes exist). The CRM's job is visibility + next-move guidance, not data entry. (see origin: artifacts/crm-design-brief.md §1–2)

- **No per-group seat caps enforcement** — show counts + Scholars warning only (Open Q1 pending).

- **Deliberate deviation from brief §5.3:** review fields (`review_status`, `review_notes`, `group_assignment`, `reviewed_by`, timestamps) live in a separate staff-only `child_reviews` table, NOT as columns on `children` — the existing `children` RLS policy is row-level `FOR ALL`, so brief-as-written would let parents write their own review status. Parent dashboard still sees progress via the `children.status` sync (Decision 6); CRM derivation reads MEMBER only from `child_reviews`. See Decision 1.
- **Accessibility baseline (deliberately minimal for two users):** focus trap + focus return on drawer/modal, Escape closes, and every kanban-drag outcome reachable via keyboard through the drawer's stamp/override buttons. Nothing beyond that in v1 — stated here so it's a decision, not a gap.

## Key Technical Decisions

1. **`child_reviews` is a separate staff-only table — NOT columns on `children`** *(deviation from brief §5.3, security-driven)*. The existing `children` RLS policy is `FOR ALL USING (auth.uid() = parent_id)`; RLS is row-level, so review columns on `children` would be parent-writable — a parent could set themselves to member. `child_reviews` (`child_id` unique FK, `review_status`, `review_notes`, `group_assignment`, `reviewed_by`, timestamps) carries staff-only RLS. Stage derivation reads MEMBER **only** from `child_reviews`; `children.status` is trusted only for DOSSIER STARTED/SUBMITTED (states parents legitimately control).
2. **Stage derivation is a shared pure TS function, not a SQL view.** `deriveStage(truth)` in `app/crm/lib/engine.ts`, unit-tested, used by table/kanban/KPIs (all render server-side from one query set). Weekly/historical funnel actuals are computed **directly from truth timestamps**, with three integrity provisions: **(a)** `deposits` gains a **`refunded_at`** column set by the `charge.refunded` webhook branch — a scoped one-line amendment to the "webhook untouched" invariant, without which refunds are unrepresentable as-of-week-end; a deposit counts in the week of `created_at` while not refunded by that week's end (net view, matching the thermometer). **(b)** The **calls rows aggregate from the immutable per-stamp events** in `family_stage_history` (Decision 13 writes one at every stamp), never from the mutable latest-wins columns — those feed only `deriveStage`. **(c)** **`signup_at` / `dossier_submitted_at` are snapshotted onto `families`** by the sync paths, and aggregation coalesces live truth rows with the snapshots, so an account deletion cannot rewrite past weeks' numbers. Aggregation **excludes events timestamped before Jul 13** (week-clamping is selector-UI behavior only; the stamp backdate field is floored at the sprint start). `family_stage_history` records **staff-driven events only** (overrides, reopen, review moves, merges, stamp events), written by the acting server action at event time. Rationale: matches alphahub's testable-pure-function posture and the repo's aggregation-in-TS pattern; avoids dual-maintenance of SQL + TS; eliminates read-path mutations.
3. **Families sync is a DB trigger on `parents` INSERT** (match-lead-by-email-then-link, else create) **plus an idempotent backfill/repair script.** `parents` rows are created client-side (AccountModal upsert), so only a trigger fires reliably. The trigger function is **`SECURITY DEFINER`** (table-owner-owned, `SET search_path = public` — the repo's `seats_claimed()` convention): it fires during a *parent-authenticated* INSERT, and with default invoker rights the CRM tables' admin-only RLS would silently block every sync. Emails compare case-insensitively (`lower(email)`; skip the citext extension — one unique index on `lower(email) WHERE email IS NOT NULL` does the job). The backfill only needs to create family rows (identity snapshot + consent mapping + `welcome_email_at` copy) — **no synthetic history**: weekly funnel actuals derive from truth timestamps directly (Decision 2), so launch-day attribution is correct by construction.
4. **`families.parent_id` is `ON DELETE SET NULL` + identity snapshot.** Sync copies name/email/phone into `families` so account deletion degrades the row to a lead with CRM history intact, and logs an "account deleted" timeline event. **Authority rule for linked families:** while `parent_id` is set, identity fields render and send from the `parents` row (the snapshot is dormant until deletion); `updateContact` is restricted to lead-only families. **Retention:** indefinite by default; a household's PII-deletion request is honored **manually via service-role SQL in v1** (run-book entry: blank snapshot fields + redact notes, log to `crm_audit_log`). A `purgeFamilyIdentity` staff action — whose real scope includes `family_notes` free text and audit-trail redaction — is **deliberately deferred** (untraced to any brief requirement; parked alongside the S10 deposits-cascade revisit). (Pre-existing landmine flagged, not fixed here: `deposits` cascade-deletes with the auth user — acceptable in test mode, **must be revisited at S10 go-live**; see Risks.)
5. **`stage_override` is void against higher truth — by construction.** `deriveStage()` itself ignores an override whenever deposit-paid or member truth is present (pure function, no mutation, no execution-point problem), so the CRM cannot disagree with `seats_claimed()` even before any cleanup runs. The UI shows the derived stage plus an "override superseded" chip; the stale override row is cleared as bookkeeping the next time a staff action touches the family (logged to stage history + audit). Drawer gets an explicit **REOPEN** action (clears override, audited) so LOST/WAITLIST are exits, not traps.
6. **MOVE CANDIDATE writes `child_reviews` and syncs `children.status` (service-role)** so the parent dashboard stepper keeps showing review progress (it already renders in_review/invited/offered/member). One enum, one spelling: reuse the existing `SeatStatus` values from `app/dashboard/data.ts` for `review_status` (`in_review`, `invited`, `offered`, `member` — not the brief's hyphenated labels; labels are display-layer).
7. **Vitest is introduced** (repo currently has zero tests). Config mirrors alphahub's (jsdom, `@` alias, tests under `app/crm/__tests__/` or root `__tests__/`), scoped to the CRM engine + validation logic. This is the brief's §12 testing posture.
8. **Staff auth = proxy (cheap JWT gate) + `requireStaff()` (authoritative).** New root `proxy.ts` — **Next 16 renamed the middleware convention to proxy** (exported function `proxy`, Node.js runtime by default, which suits the Supabase session check; verified in `node_modules/next/dist/docs/01-app/03-api-reference/03-file-conventions/proxy.md`) — matches `/crm/:path*`, checks `app_metadata.role === 'admin'` from the session JWT, rewrites non-staff to the 404-style screen. Every server component/action additionally calls `requireStaff()` which verifies the `staff` row `is_active` — **read via `supabaseAdmin()`, not the session client**, so its verdict never depends on policy shape. RLS: the `staff` table itself gets **only** the JWT-role check (a self-referencing EXISTS triggers Postgres's "infinite recursion detected in policy" and would break every CRM query); all **other** CRM tables use `(auth.jwt() -> 'app_metadata' ->> 'role') = 'admin' AND is_active_staff()`, where `is_active_staff()` is a tiny `SECURITY DEFINER` helper wrapping `EXISTS (select 1 from staff where id = auth.uid() and is_active)` — fail-closed, and revocation bites at the PostgREST layer too, not only inside `requireStaff()`. No UPDATE/DELETE policies on `crm_audit_log` + immutability trigger (alphahub 005 pattern).
9. **Consent lifecycle is complete in schema from day one:** `consent_given`, `consent_at`, `consent_source`, `consent_revoked_at`. Revoke is a staff action (audited). Merge rule: consent = OR of sources, keep earliest `consent_at`, **a revocation is never resurrected by a merge**. The gate applies to email sends *and* "mark as sent elsewhere" (CASL covers texts). Composer emails include an unsubscribe/identification footer (CASL CEM requirement).
10. **Sends log only after Resend accepts** (the `/api/welcome` pattern): no `library_sends` row, no `last_touch_at`, no timeline event on failure — prevents phantom "concern addressed" co-pilot state.
11. **All week math goes through one `weekOf()` helper pinned to America/Toronto** (sprint window Jul 13–Sep 4; W8 is a partial week ending Sep 4 — clamp the selector to W1–W8 and show a "sprint ended" state after).
12. **CRM color tokens are additive.** The brief's `#0300ED` / `#ECEAE5` / etc. become CRM-scoped tokens in `app/globals.css`; the marketing site's existing `#22219b` blue is untouched.
13. **Call stamps are editable, not one-way.** `LOG CALL BOOKED/HELD` sets the timestamp; a second tap offers overwrite (latest wins, each stamp logged in timeline); a clear affordance exists (audited). Multiple calls beyond the first are timeline events, not schema.
14. **Read audits are scoped to dossier detail** (`drill-down` action on opening a child's dossier, per brief §12 / alphahub) — not on every pipeline read (two users, noise).

## High-Level Technical Design

> *This illustrates the intended approach and is directional guidance for review, not implementation specification. The implementing agent should treat it as context, not code to reproduce.*

**Stage derivation (pure function, first match wins):**

```
deriveStage({ override, reviews, deposits, calls, children, parentId }):
  override set AND no higher truth      → LOST | WAITLIST     (override is void when deposit-paid/member truth exists — Decision 5)
  any child_reviews.review_status = member → MEMBER
  any deposit status = paid             → DEPOSIT_PAID        (refund webhook flips the row → derivation demotes)
  call_held_at                          → CALL_HELD
  call_booked_at                        → CALL_BOOKED
  any children.status ≠ draft           → DOSSIER_SUBMITTED
  any children row                      → DOSSIER_STARTED
  parent_id present                     → ACCOUNT_CREATED
  else                                  → INTERESTED
```

**Data flow (sync contract §10):**

```mermaid
flowchart LR
  A[AccountModal signup] -->|parents INSERT| T[families sync trigger]
  T -->|match lead by lower(email) → link, else create| F[(families)]
  W[Stripe webhook] --> D[(deposits)]
  Q[Dossier queue action] --> CR[(child_reviews)] -->|service-role sync| CS[children.status]
  S[Staff stamps/overrides/notes/sends] --> F
  F & D & CR & CH[(children)] --> E[deriveStage / dashboards / KPIs]
  E --> H[(family_stage_history)] 
  S --> AU[(crm_audit_log)]
```

Stage *history* (`family_stage_history`) records staff-driven events only, written by the acting server action at event time. System-driven transitions (signup, dossier, deposit) need no history rows at all — weekly funnel actuals and timeline events derive **directly from truth timestamps** (Decision 2), so nothing depends on when staff happened to look.

## System-Wide Impact

- **Interaction graph:** the families sync trigger fires on every parent signup (AccountModal path unchanged); MOVE CANDIDATE now writes `children.status` (previously parent-only writes — parent dashboard rendering already supports all values); the Stripe webhook gains exactly one line (`refunded_at` stamp in the `charge.refunded` branch — Decision 2a) and its rows now drive CRM stage derivation; `/api/welcome` gains one line (stamps `families.welcome_email_at` on successful send).
- **Error propagation:** server actions return `{success,error}` (never throw to client); Resend failures never create send state; trigger failures on signup must not block account creation (trigger is AFTER INSERT and defensive — exceptions swallowed; the Unit 6 standing sync-health stat makes silent failure visible, and the backfill script is the repair).
- **State lifecycle risks:** account deletion (families SET NULL + snapshot); deposit refunds (derivation demotes + history logs); override vs truth conflicts (auto-clear); duplicate families (merge action). The pre-existing deposits-cascade-on-user-delete is *flagged, not changed*.
- **API surface parity:** none of the public API routes change; `/crm` adds no parent-visible endpoints; CRM fields never serialize into parent-accessible responses (RLS + service-role-only reads enforce this).
- **Integration coverage:** signup→family sync, webhook→stage demotion, review→parent-dashboard sync, send→co-pilot input are all cross-layer paths with explicit integration scenarios in Units 2, 4, 5, 7.
- **Unchanged invariants:** parent RLS policies, `seats_claimed()`, the checkout route, marketing-site tokens (`#22219b` stays), the parent dashboard UX (except the review-status stepper now advancing — which it was built to display). The webhook and welcome routes each change by exactly one additive line (`refunded_at`, `welcome_email_at`) — behavior otherwise untouched.
